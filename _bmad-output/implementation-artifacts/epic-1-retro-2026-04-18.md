## What Didn't Go Well

### 1. Upstream ownership not retained across sessions
Bogdan controls the upstream chess libraries and the foolnotion Nix overlay. He communicated this multiple times during the epic. Agents still invented CMake workarounds for packaging problems (Story 1.2: downstream `find_dependency` workarounds) rather than surfacing the issue to Bogdan and waiting for an upstream fix. This was wasted effort and sometimes made things worse.

**Root cause:** Instructions communicated in one session don't persist to the next agent session unless written to a durable, always-loaded file.

### 2. Source code before documentation
Agents grepped DuckDB and SQLite headers to understand APIs instead of reading project documentation first. This led to the Story 1.5 Clang 21 / DuckDB C++ API incompatibility being discovered mid-implementation rather than during planning. Reading the DuckDB docs would have revealed the C API option earlier.

### 3. vcpkg deferred too late
Bogdan had to explicitly push back on mid-story vcpkg additions while Nix packaging issues were already in flight. Mixing two packaging systems' concerns simultaneously added noise to already complex stories.

### 4. Review verification gap
Story 1.4's five code review patches were applied but could not be build-verified (`cmake` unavailable in the review agent's environment). The fixes are believed correct but were not proven by a passing test run.

### 5. Style inconsistencies across stories
Conventions that were never written down drifted across the 5-story span:
- `k_` prefix for constants appeared (never requested)
- SQL expressed as multiline raw strings in one file, `constexpr char const*` in another
- These are small but represent the broader problem: unwritten conventions diverge silently across fresh agent contexts

### 6. Multi-tool retention gap
The project uses Claude, Copilot, and Copilot in OpenCode. Durable rules written only to `CLAUDE.md` don't reach the other two tools. No single authoritative conventions file exists that all three tools read.

## Key Technical Discoveries

### DuckDB C++ API incompatible with Clang 21
DuckDB's `profiling_utils.hpp` uses `unique_ptr<ActiveTimer>` before `ActiveTimer` is fully defined. Clang 21 rejects this with "invalid application of sizeof to an incomplete type." Marking headers SYSTEM does not help — Clang still instantiates the templates.

**Resolution:** All DuckDB code in `motif_db` uses the pure C API (`<duckdb.h>`) — `duckdb_database`, `duckdb_connection`, `duckdb_appender`. No C++ API (`duckdb::DuckDB`, `duckdb::Connection`, `duckdb::Appender`).

**Impact on Epic 2:** Any new DuckDB-touching code must also use the C API.

### `game_store::insert` duplicate policy
The `insert_options`/`skip_duplicates` struct was removed during Story 1.3 review. `insert()` always returns `error_code::duplicate` on conflict. Callers (including the Epic 2 import worker) must handle duplicate results explicitly — treat as skip-and-log, not fatal error.

### Architecture document describes wrong DuckDB API
The architecture document's DuckDB section describes the C++ API surface. This is now incorrect. Needs updating before Epic 2 agents consult it.
