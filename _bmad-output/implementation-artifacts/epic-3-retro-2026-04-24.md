## What Went Well

1. **Layered architecture held perfectly** — `position_search` → `opening_stats` → `opening_tree` built cleanly on each other with zero boundary violations. `motif_search` never included DuckDB or SQLite headers. All storage access routed through `motif_db` public APIs.

2. **Agent portability proven** — Story 3.3 used claude-sonnet-4-6 (different model from 3.1 and 3.2) and still correctly applied carry-forward patterns: dedup keys, hash combiners, orphan handling, opening name selection. Story specs + CONVENTIONS.md are mature enough for model switching.

4. **123 tests, zero sanitizer violations, zero warnings** — Clean across all three stories under both `dev` and `dev-sanitize` builds. Build-type-aware perf test pattern (enforce on release only, warn on dev, skip on sanitize) works correctly.

5. **Per-story carry-forward notes are a breakthrough** — Story 3.3 explicitly listed mandatory carry-forwards from 3.1 and 3.2 (dedup pattern, orphan handling, uint16 accessors, is_sanitized_build helper). This prevented agent mistakes across sessions.

## What Didn't Go Well

### 1. Story 3.2 scope expansion
Story 3.2 (Opening Statistics) expanded well beyond its stated scope. In addition to the opening stats feature, it also: recalibrated import perf tests across the entire suite, added a `perf-release` CMake preset, removed two entire import strategies (direct-write and partitioned-rebuild), and benchmarked the full TWIC corpus (~3.41M games).

The work was valuable and necessary — but packaging it in one story made review harder (3 separate review passes with different scopes) and violated the "one logical unit" principle.

**Root cause:** Pre-existing import perf test failures (dev build hitting release-calibrated thresholds) blocked the full test suite. Fixing them was prerequisite to marking 3.2 done.

**Resolution:** Perf work that touches other modules deserves its own story boundary. "Adjacent fixes" are acceptable; cross-module restructuring is not.

### 2. Deferred items doubled (8 → 16)
The deferred-work list grew from 8 items at Epic 2 close to 16 after Epic 3. Several items have been re-flagged across multiple stories:
- DuckDB prepared statements: flagged in devlog W18, deferred in 3.1, deferred again in 3.2
- Import config validation: deferred in Epic 2, deferred again in 3.2
- Missing positions.duckdb detection: deferred in 3.1, not addressed since

**Root cause:** Items aren't blocking correctness, so they keep getting postponed. But accumulation creates review fatigue and increases migration cost (every new query method makes the prepared statement migration harder).

**Resolution:** "Defer twice, then it gets a story" rule — if an item is deferred more than twice, it gets promoted to a dedicated story.

### 3. NFR02 gap on high-fanout positions
`opening_stats::query` P99 reaches 700ms on the 1M corpus for popular positions (e.g., after 1.e4), exceeding the 500ms AC1 target by 200ms. The bottleneck is per-game SQLite context fetches for ECO/opening_name/moves.

**Fix path:** Extract only `moves[2*ply]` from moves blob instead of full deserialization, or cache game contexts per query. Requires `position_store` changes.

### 4. Commit frequency still suboptimal
Epic 2 retro flagged commit cadence. Epic 3 improved but still produced mega-commits, particularly in Story 3.2. The team agreed: commit per logical unit (bug fix, test addition, refactoring, feature) — not per story or per session.

## Key Technical Discoveries

### Single DuckDB self-JOIN query for tree prefetch
Story 3.3's `query_tree_slice` uses a self-JOIN on the `position` table to fetch all descendants within `max_depth` in a single query. This satisfies AR10's requirement for "one DuckDB query" for the prefetch phase and is significantly more efficient than N queries per node.

### `opening_stats::continuation` extended with `result_hash`
Story 3.3 added `result_hash` to the `continuation` struct, populated via chesslib replay during aggregation. This enables `opening_tree::expand` to populate child nodes without re-querying. The extension was planned from Story 3.2's design — good forward-planning.

### Import pipeline simplified to single strategy
Story 3.2 removed the direct-write and partitioned-rebuild import strategies after benchmarking confirmed the default deferred-rebuild path is optimal. The codebase is now simpler with one import path.

### Build preset architecture matured
Added `perf-release` preset. Perf tests now: enforced on release builds, warned on dev builds, skipped entirely on sanitizer builds. `is_sanitized_build` helper extracted to `test/source/test_helpers.hpp` for reuse.

## Epic 4b: HTTP API Layer (New)

**Rationale:** An HTTP API layer between motif-chess and any future frontend (Qt, web, or experimental) provides a low-stakes experimentation space for features too risky or speculative for motif-chess proper. The OpenAPI spec becomes a versioned interface boundary.

**Proposed Stories:**
1. cpp-httplib dependency + HTTP server scaffold (incl. CORS, DB config at startup, health endpoint)
2. Position search endpoint
3. Opening stats endpoint
4. Import trigger + SSE progress stream
5. Game list endpoint
6. Single game retrieval endpoint (GET /games/{id})

**Deliverable:** Incremental OpenAPI spec (built per story), consolidated as the epic's exit artifact.

**Dependencies:** All met — `position_search`, `opening_stats`, `opening_tree`, `import_pipeline`, `database_manager` are stable and tested.
