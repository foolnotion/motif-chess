## Key Technical Discoveries

### Serial import + deferred rebuild is the winning path

Per-game position writes to DuckDB are both slow and memory-intensive. The fast path is:
1. Import games serially with `write_positions=false` (SQLite only)
2. Rebuild position store in bulk: `rebuild_position_store(create_index=true, sort_by_zobrist=false)`
3. Result: ~3.4M games in ~6 seconds

This avoids the previous issues (slow per-game DuckDB inserts + high memory from accumulated position rows).

### The remaining frontier is concurrency

The import path is now fully serial and very fast on the serial bottleneck (SQLite single-writer). The next performance gain comes from exploring concurrency: batch handoff, WAL mode, write-behind, or re-enabling the parallel compute stage.

### `count_games` is a heuristic

Counts `[Event "` markers without PGN structural awareness. Acceptable for progress bars, not authoritative.

### `import_progress` now mirrors `import_summary`

Both expose `errors` and `total_games`, closing an observability gap caught in review.
