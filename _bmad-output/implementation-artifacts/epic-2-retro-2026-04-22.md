## What Went Well

1. **CONVENTIONS.md worked** — Consistency improved noticeably over Epic 1. Agents followed rules.

2. **3-stage pipeline architecture proved extensible** — SERIAL reader → PARALLEL compute → SERIAL writer accommodated every change (error counting, logging enrichment, SQLite batching, position rebuild config) without restructuring.

3. **Code reviews were substantive** — 23+ findings across Stories 2.4 and 2.5 alone. The adversarial review layers (Blind Hunter, Edge Case Hunter, Acceptance Auditor) caught real issues: counter gaps, non-diagnostic logging, missing API fields.

4. **Profiling-first optimization was excellent** — We measured, then decided. Every optimization was grounded in `perf stat`, `perf record`, and `strace -c` evidence. This discipline produced the two biggest wins:
   - **Deferred position rebuild** — `write_positions=false` + bulk `rebuild_position_store()` avoids per-game DuckDB overhead and memory blowup. ~3.4M games in ~6 minutes.
   - **[D009] Sorted-by-zobrist beats ART** — 4x better P99, 8x less memory. Discovered through parameterized benchmarking in Story 2.8.

5. **Serial-first benchmarking isolated bottlenecks** — Defaulting to `num_workers=1`/`k_default_lines=1` during 2.5-2.9 made profiling results reproducible and eliminated confounding variables.

6. **Zero sanitizer violations throughout** — All stories passed `dev-sanitize` clean. No late-discovered memory or undefined-behavior issues.

## What Didn't Go Well

### 1. Session limit interruptions left incomplete code marked as done
Stories 2.5 and 2.8 were most affected. The code was not sloppy by design — agents were cut off mid-work by session limits, leaving half-finished changes committed as complete. Story 2.8's 28 code review patches (including compilation errors, race conditions, uninitialized variables) were the result of interrupted work, not careless implementation.

**Root cause:** Session boundaries are a hard constraint. When an agent hits the limit, there is no opportunity to compile, test, or clean up.

**Resolution:** This is an individual responsibility — preemptive wrap-up before hitting usage limits. No process band-aid will fix it.

### 2. Sprint-status and epics file fell out of sync mid-sprint
Stories 2-6 and 2-7 had spec files but weren't tracked. Stories 2-8 and 2-9 were tracked but without full slugs. The sprint change proposal from 2026-04-20 was never fully applied. The epics file stopped being authoritative mid-sprint.

### 3. Config defaults changed silently for benchmarking
`num_workers` and `k_default_lines` were set to 1 for serial benchmarking isolation. Not reverted or documented in a story. Code review caught it, but it illustrates the risk of unscoped configuration changes.

### 4. Deferred items grew faster than resolved
8 deferred items accumulated across code reviews and story work. None from Epic 1's retro were explicitly closed — they were absorbed by CONVENTIONS.md or made moot by implementation, but never formally marked done.

### 5. Helper logic duplicated across translation units
`find_tag`, `parse_elo`, `pgn_result_to_string`, `pgn_result_to_int8` were reimplemented in both `import_worker.cpp` and `import_pipeline.cpp` anonymous namespaces. Identical logic, not shared. Maintenance risk.

### 6. Multi-agent style drift
Different agents across different sessions introduced minor convention violations (e.g., `k_` prefix for constants despite CONVENTIONS.md explicitly prohibiting it). Manual code inspection still needed to catch these.

### 7. Commit cadence too slow
Changes accumulated locally across entire stories or sessions before being committed. Mega-commits with 6+ concerns make bisection, rollback, and review harder. Related to the session-limit problem — uncommitted work is at risk of loss.

## Key Technical Discoveries

### Serial import + deferred rebuild is the winning path
Per-game position writes to DuckDB are both slow and memory-intensive. The fast path is:
1. Import games with `write_positions=false` (SQLite only)
2. Rebuild position store in bulk: `rebuild_position_store(create_index=true, sort_by_zobrist=false)`
3. Result: ~3.4M games in ~6 minutes; 1M rebuild-only in ~2 minutes

### [D009] Drop ART index in favor of sorted-by-zobrist
Benchmarked on 3.4M games (288M position rows, 200 sampled hashes):

| Variant | P50 | P99 | Total | RSS |
|---|---|---|---|---|
| ART indexed | 630 us | 224,278 us | 2,428 ms | ~24 GB |
| No index | 50,413 us | 129,151 us | 10,770 ms | ~3 GB |
| **Sorted no index** | **1,254 us** | **58,719 us** | **608 ms** | **~3 GB** |

Sorted zonemaps give near-index P50, 4x better P99, and 8x less memory.

### The remaining frontier is concurrency
The import path is fully serial. The next performance gain comes from exploring: batch handoff, WAL tuning, write-behind, or re-enabling the parallel compute stage.

### `position_store::insert_batch` already uses DuckDB C API appender
This was confirmed in Story 2.8 — not the bottleneck. Per-row INSERT statements are not in use.

### SQLite WAL is NOT the I/O bottleneck
`strace -c` showed only 6 fsync calls for 10k games. WAL mode batches I/O effectively.

### Futex contention is the pipeline friction
~8% of wall time in `futex` (108 calls, 916ms) — per-game token handoff in the taskflow pipeline creates constant thread wake/sleep cycles.

### Story 2.7 dropped — NFR16 still depends on full-materialization
`pgnlib::import_stream` silently drops variations, NAGs, and annotations. It solves the wrong problem. If the parser proves a real bottleneck, the correct approach is a lexy callback-based action grammar, developed independently.
