# Week 17 — 2026-04-20

## Decisions

- **[D001] Add Epic 2 performance track (stories 2-6, 2-7, 2-8):** During story 2-5
  implementation and profiling, three distinct bottleneck layers were identified: upstream
  chesslib SAN resolution, pgnlib PGN materialization, and the downstream SQLite/DuckDB
  storage path. Rather than absorbing this into the current story or opening a new epic,
  a sprint change proposal was approved to extend Epic 2 with three focused stories.
  Rejected: rolling back completed work; creating a new epic (premature); allocator-first
  redesign (not yet justified without upstream wins first).

- **[D002] Drop story 2-7 (pgnlib import_stream integration):** `pgnlib::import_stream`
  silently discards variations, NAGs, and annotations — it solves the wrong problem for a
  database that must preserve full PGN fidelity. If the parser ever proves a real
  bottleneck, the correct approach is a lexy callback-based action grammar, developed
  independently outside the sprint. Rejected: proceeding with import_stream (data loss);
  immediate lexy grammar work (premature without profiling evidence).

- **[D003] DuckDB C API only (`<duckdb.h>`) — no C++ API:** DuckDB's own documentation
  marks the C++ API as internal: "It is not guaranteed to be stable and can change without
  notice. If you would like to build an application on DuckDB, we recommend using the C
  API." The Clang 21 incompatibility discovered in Epic 1 (incomplete type in
  `profiling_utils.hpp`) is a consequence of using an unstable internal API. All DuckDB
  code uses `<duckdb.h>` exclusively (`duckdb_database`, `duckdb_connection`,
  `duckdb_appender`). This applies to all future stories. Rejected: C++ API (explicitly
  unsupported for applications); C++ wrapper shim (unnecessary indirection over a stable
  C interface).

- **[D004] Use legality-preserving stateful SAN replay in import paths:** The direct
  `chesslib` vs `chess-library` SAN corpus benchmark initially showed a large replay-path
  speedup, but that result was invalid because the new `san::replayer` fast path skipped
  legality checking for single-candidate moves. After restoring legality checks upstream,
  the stateful path still matched `chesslib::san::from_string` on final positions and
  remained slightly faster on the 10k corpus (`453ms` vs `465ms` with verification on 5
  rounds). Because PGN import naturally consumes consecutive SAN moves on a rolling board,
  motif-chess should use the stateful API in its import loops before any further full-path
  profiling. Rejected: keeping the stateless parser in import despite a matching stateful
  API; treating the pre-fix replay benchmark as valid optimization evidence.

## Learned

- **NFR03 gap is 11×:** Post-2.6 baselines on 10k games: SAN-only `0.78s`, full import
  with deferred position index `13.48s`. Extrapolating to 10M games gives ~3.7 hours
  against a 20-minute target. Since position indexing is deferred in the slow baseline,
  the bottleneck lives in the SQLite game-store write path, not DuckDB.

- **DuckDB Appender is the key insert primitive at scale:** [aix](https://github.com/thomas-daniels/aix)
  (DuckDB chess extension over 7 billion Lichess games) uses `duckdb_appender` for all
  bulk inserts. Per-row SQL `INSERT` would be orders of magnitude slower. If
  `position_store::insert_batch` is not already using the appender, that is the prime
  Task 3 candidate for story 2-8.

- **scoutfish confirms the lexy direction:** [scoutfish](https://github.com/mcostalba/scoutfish)
  parses PGN with a two-table FSM (`ToToken[256]`, `ToStep[11][16]`) — one byte per table
  lookup, no heap allocation, L1-resident. This is exactly the callback-based action
  grammar approach for any future parser work. scoutfish's 16-bit move encoding is also
  identical to the CPW convention used by `chesslib::codec` — no change needed there.

- **Parallel byte-range scan eliminates a position index at moderate scale:** aix runs
  full-database heatmap queries over 7 billion games in 92–241 seconds on 24 threads with
  no pre-built position index. At 10M games that scales to ~130–340ms per query — well
  within NFR01's 100ms P99 budget if we use parallel scanning rather than a Zobrist hash
  index. Worth validating against our DuckDB schema before investing in index tuning.

- **`game_store::insert` always returns `error_code::duplicate` on conflict** — the
  `skip_duplicates` option struct was removed in the 1.3 code review. All callers,
  including the import worker, must treat duplicate as skip-and-log, not fatal.

- **`CONVENTIONS.md` is the cross-tool authority:** Epic 1 retro identified that rules
  written only to `CLAUDE.md` do not reach Copilot or OpenCode agents. `CONVENTIONS.md`
  is the single source of truth; all tool config files reference it.

- **The large replay benchmark win was a correctness bug, not a real optimization:** Once
  legality checks were restored in `chesslib::san::replayer`, the SAN-only gap between the
  stateful and stateless `chesslib` paths shrank to noise-level territory. The remaining
  meaningful performance gap is versus `chess-library`, which points to representation-level
  limits in the current `0x88` design rather than an unexploited parser shortcut.

## Problems and Solutions

- **Problem:** pgnlib `import_stream` loses variations, NAGs, and annotations — integrating
  it into the pipeline would silently degrade PGN fidelity with no visible error.
  **Solution:** dropped story 2-7; documented the lexy callback-based grammar as the
  correct future approach if profiling ever shows the parser as a bottleneck.

- **Problem:** Story 2-8 initially included a `duckdb::Appender` C++ code example, which
  is invalid under Clang 21. **Solution:** corrected to `duckdb_appender` C API pattern
  before the story was handed off.

- **Problem:** NFR03 (10M games in <20 minutes) appears unreachable at current throughput
  (11× gap). **Solution:** story 2-8 profiles to find the dominant bottleneck before
  committing to a fix; prime hypothesis is that `position_store::insert_batch` is using
  individual SQL INSERTs rather than `duckdb_appender`.
