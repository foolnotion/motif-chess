# Week 17 — 2026-04-25

## Decisions

- **[D018] HTTP server as a separate `motif_http` binary:** The HTTP API layer runs as its own executable (`motif_http_server`) linked against `motif_http` (static lib), `motif_db`, `motif_search`, and cpp-httplib. The server uses pImpl to confine httplib's platform headers to `server.cpp`. `motif_search` is a PRIVATE dependency of `motif_http` — the HTTP module never reaches into `position_store` directly.

- **[D019] DB-level pagination, not HTTP-layer slicing:** A common opening position can have millions of matches. Materializing all rows before slicing at the HTTP layer would blow RAM and violate NFR01 (100ms P99). `LIMIT/OFFSET` lives in the DuckDB query so only the requested page is ever materialized. Default limit is 100 rows (~5 KB JSON); max clamped to 500 (~25 KB).

- **[D020] `ORDER BY game_id, ply` for stable pagination:** Without an explicit ORDER BY, DuckDB can return rows in any physical order across pages, causing duplicates or gaps. Added stable ordering to `query_by_zobrist` whenever `LIMIT`/`OFFSET` is present.

- **[D021] `limit=0` treated as "return empty array":** The story spec says "at most `limit` items are returned". `limit=0` returning all rows was a footgun. Changed the endpoint to short-circuit: if `limit` is explicitly `0`, return `[]` immediately without querying.

- **[D022] `perf-release` preset committed to project presets:** Moved from `.gitignore`d `CMakeUserPresets.json` into `CMakePresets.json`. The preset is a project concern — every developer and CI needs it to run performance tests. `CMakeUserPresets.json` reduced to an empty shell for machine-specific overrides only.

- **[D023] `httplib` added to Nix dev shell:** The `flake.nix` package already listed `httplib` in `buildInputs`, but the dev shell only inherited `nativeBuildInputs` + `buildInputs` and missed propagating httplib's CMake config. Fresh configure presets like `perf-release` couldn't find it. Added `httplib` explicitly to the dev shell's `buildInputs`.

## Learned

- **cpp-httplib route pattern `:param` does not match empty segments.** `/api/positions/:zobrist_hash` does not fire for `/api/positions/` — an empty path segment. The empty-hash AC4 case required separate explicit routes for `/api/positions` and `/api/positions/`.

- **`std::from_chars` with `std::size_t` naturally rejects negative values** on platforms where `size_t` is unsigned — the parse fails, which is exactly what AC5 requires for negative `offset`/`limit` parameters.

- **DuckDB `LIMIT 0` means "return zero rows"**, so the pagination guard must be `limit > 0` to append the clause. When `limit == 0` (the default "no pagination" value), the query returns all matching rows — which is correct for internal callers like `opening_stats` but wrong for the HTTP endpoint where `?limit=0` should return `[]`.

- **Glaze reflection requires named namespaces.** Anonymous-namespace structs don't have external linkage, so `glz::write_json` can't find their metadata. The `health_response` and `error_response` structs must live in `motif::http::detail`.

## Problems and Solutions

- **Problem:** `limit=0` in the endpoint bypassed pagination and returned all matches.
  **Solution:** Added a guard in the route handler: if the client explicitly passed `limit=0`, return `[]` immediately.

- **Problem:** Paginated queries had no `ORDER BY`, so `LIMIT/OFFSET` pages could duplicate or skip rows across requests.
  **Solution:** Appended `ORDER BY game_id, ply` to the DuckDB query when `limit > 0`.

- **Problem:** `/api/positions/` (empty hash) didn't match the `:zobrist_hash` route parameter, returning 404 instead of 400.
  **Solution:** Registered explicit handlers for `/api/positions` and `/api/positions/` that return the AC4 error.

- **Problem:** `perf-release` CMake preset was `.gitignore`d, so new clones lacked it and CI couldn't run performance tests.
  **Solution:** Moved all project-level presets into `CMakePresets.json`; reduced `CMakeUserPresets.json` to a stub.

- **Problem:** Fresh CMake configures (e.g., `perf-release`) failed to find `httplib` because the Nix dev shell didn't propagate its CMake config.
  **Solution:** Added `httplib` to `buildInputs` in the dev shell section of `flake.nix`.

## Stories 4b.3–4b.6 (continued, same week)

### Decisions

- **[D024] Glaze meta specialization for non-aggregate response types:** `opening_stats::stats` has a user-declared default constructor (`= default`), which makes it a non-aggregate in C++23, preventing glaze auto-reflection. Added an explicit `glz::meta<motif::search::opening_stats::stats>` specialization in `server.cpp`. Rule: any response type with a user-declared constructor needs a `glz::meta` — plain aggregates do not.

- **[D025] `database_mutex` added to `server::impl` for all DB-backed routes:** Concurrent requests could reach `database_manager` from different httplib threads. Added `std::mutex database_mutex` to `server::impl` and wrapped every DB call site in a local `std::scoped_lock`. `sessions_mutex` added separately for the sessions map, keeping DB and session locking orthogonal.

- **[D026] `server::impl` promoted from anonymous struct to full pImpl with `setup_routes()`:** The original `impl` was a minimal anonymous struct. Promoted it to a declared type with a proper destructor, deleted copy/move, and a `setup_routes()` member function. Route lambdas now capture `this` instead of `&dbmgr`, giving them access to both the mutex and the sessions map without threading extra parameters.

- **[D027] SSE via `set_chunked_content_provider` with poll-sleep loop:** httplib's chunked content provider is the only push mechanism without additional threading infrastructure. The callback returns `true` to continue and `false` to close the stream. A 250ms sleep between polls keeps CPU near zero during idle imports. The tradeoff (blocks one httplib thread per active SSE client) is acceptable given the single-active-import constraint.

- **[D028] Single active import enforced with HTTP 409 Conflict:** Multiple concurrent imports would contend on SQLite writes and checkpoint state. A `std::ranges::any_of` over the sessions map under `sessions_mutex` rejects a second `POST /api/imports` while any session has `done == false`. Rejected with 409 and `{"error":"import already running"}`.

- **[D029] `uint64_t` intermediate in `parse_game_id` for overflow detection:** `std::from_chars<uint32_t>` returns `errc::result_out_of_range` for overflow but gives no way to distinguish "value 0" from "parse failed". Parsing into `uint64_t` first, then explicitly rejecting `value == 0` and `value > UINT32_MAX`, makes each rejection reason explicit and independent. Same pattern now used consistently for game IDs and Zobrist hashes.

- **[D030] DB-level pagination in `list_games` via `LIMIT/OFFSET` in SQL:** Materializing all game rows before slicing at the HTTP layer would be wasteful for large databases. `game_store::list_games` accepts a `game_list_query` struct carrying `limit`, `offset`, and optional `player`/`result` filters; the SQL applies `ORDER BY g.id ASC LIMIT ? OFFSET ?` directly. Default limit 50, max clamped to 200 at the HTTP layer.

### Learned

- **Glaze reflection fails silently for anonymous-namespace types and non-aggregates.** Two separate failure modes: (1) anonymous-namespace types lack external linkage so glaze can't find their metadata — all response DTOs must live in a named namespace; (2) user-declared constructors (even `= default`) make a type a non-aggregate in C++23, also defeating auto-reflection. Both require explicit `glz::meta`.

- **httplib `:param` path parameters don't match empty segments.** `/api/games/:id` does not fire for `/api/games/` — the empty segment simply produces a 404. AC requiring "bare path returns 400" must be handled by a separately registered `/api/games/` route.

- **`[[maybe_unused]]` on `glz::write_json` errors masks real failures.** If serialization fails, the response body is empty while the status is 200 OK. The pattern was pre-existing across the codebase but the 4b.6 code review caught it in the new route; the error is now checked and mapped to 500.

- **Magic-number ELO constants in test fixtures trigger clang-tidy warnings.** `insert_detailed_http_game` initially embedded `2865` and `2792` inline in struct initializers. Extracted to named `constexpr` locals (`white_elo`, `black_elo`) to keep the zero-warnings gate green.

### Problems and Solutions

## Epic 4b/4c Closeout — 2026-04-26

### Decisions

- **[D031] OpenAPI 3.1 is the HTTP API exit artifact:** `docs/api/openapi.yaml` now documents all eight HTTP routes with reusable component schemas, examples, and SSE documented pragmatically as `text/event-stream`. Rejected: leaving route contracts only in story notes, because future frontends need a machine-readable boundary.

- **[D032] Externally exposed hashes serialize as JSON strings:** Zobrist and resulting-position hashes remain `std::uint64_t` inside C++ modules, but HTTP response DTOs convert them to decimal strings before JSON serialization. Rejected: exposing 64-bit hashes as JSON numbers, because JavaScript clients cannot represent all values exactly.

- **[D033] `fmt` owns new formatting and console output:** New string construction uses `fmt::format`; new stdout/stderr output uses `fmt::print`. Rejected: ad hoc `std::cout`, `std::cerr`, `std::ostringstream`, and `std::to_string` in touched HTTP code because representation drift matters for wire contracts.

- **[D034] Import/SSE lifecycle must be explicit and bounded:** Import sessions and worker threads now have explicit cleanup, JSON-safe SSE errors, condition-variable wakeups, documented atomic ordering, defensive null handling, and destructor join ordering. Rejected: relying on implicit `jthread` destruction and unbounded completed-session retention.

### Learned

- **C++ type fidelity is not an API contract.** A field can be correct as `std::uint64_t` internally and still be wrong as a JSON number for frontend clients.

- **SSE endpoints need lifecycle design, not just event formatting.** Browser-style reconnects and long-lived streams make worker ownership, session pruning, wakeups, and timeout behavior part of correctness.

- **OpenAPI is easier to keep correct when updated with implementation.** Consolidating the spec at the end worked for Epic 4b, but Epic 4c showed that implementation, tests, and OpenAPI should move together for future API changes.
