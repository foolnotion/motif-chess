# Week 19 — 2026-04-30

## Decisions

- **[D040] Storage optimization: moves_hash index, zstd compression, DuckDB checkpoint.**
  Schema v2→v3 replaces the full `moves` BLOB in the unique dedup index with an 8-byte xxhash64
  hash (`moves_hash`), compresses the `moves` BLOB with zstd level 3, and adds `CHECKPOINT` after
  `sort_by_zobrist()` to reclaim DuckDB free blocks. Hash collisions are detected by comparing
  decompressed moves on the constraint-violation path, with up to 4 rehash retries. Existing v2
  databases auto-migrate on `open()`. Estimated savings: ~250 MB from the index, ~50-80 MB from
  compression, ~1.7 GB from DuckDB compaction. Rejected: wider hash (128-bit, overkill); storing
  moves uncompressed in the index (status quo, the problem we solved).

## Learned

- **xxHash and zstd are already in flake.nix but not wired into CMake.** Both were pre-positioned
  as Nix buildInputs but had no `find_package`, no `target_link_libraries`, and no source includes.
  Adding them to vcpkg.json is needed for Windows/vcpkg parity (per CONVENTIONS.md: both files must
  be updated for any new dependency).

- **DuckDB `CHECKPOINT` cannot run inside a transaction.** The initial attempt embedded it in the
  `sort_by_zobrist` SQL string, which runs inside the `BEGIN`/`COMMIT` block of
  `rebuild_position_store`. Moved to a separate `position_store::checkpoint()` call after COMMIT.

- **zstd's `ZSTD_getFrameContentSize()` enables decompression without knowing the output size.**
  The compressed frame header stores the original content size (enabled by default), so
  `decompress_moves` can determine the correct allocation without a separate size field.

- **SQLite `substr()` on a BLOB doesn't work with compressed data.** The `get_continuation_contexts`
  query used `substr(g.moves, ply*2+1, 2)` to extract a 2-byte move from the raw blob. With zstd
  compression this is meaningless — had to change to reading the full blob and extracting in C++.

## Done

- **Storage optimization implemented and committed** (`feat/storage-optimization` branch):
  - `moves_codec` module: `compute_moves_hash` (xxhash64), `compress_moves` (zstd-3), `decompress_moves`
  - Schema v3: `moves_hash BLOB NOT NULL`, unique index on `moves_hash` instead of `moves`
  - `game_store::insert()`: hash + compress + collision detection with rehash
  - All read paths (`get`, `get_opening_context`, `get_game_contexts`, `get_continuation_contexts`):
    decompress moves blob before returning
  - `game_store::migrate_v2_to_v3()`: backfill hash + compress existing rows
  - `position_store::checkpoint()`: reclaim free blocks after sort
  - `database_manager::open()`: calls data migration after DDL migration
  - Build system: xxhash + zstd added to vcpkg.json and CMakeLists.txt
  - All 258 tests pass on dev and dev-sanitize, zero ASan/UBSan violations

## Problems and Solutions

- **Problem:** zstd CMake target `zstd::libzstd_shared` was already defined by cpptrace's
  `Findzstd.cmake`, causing `find_package(zstd CONFIG REQUIRED)` to fail with "Some targets already
  defined."
  **Solution:** Check for existing targets before calling `find_package`, link whichever is available.

- **Problem:** `game_store::insert()` cognitive complexity exceeded 25 after adding collision
  detection loop. (clang-tidy warning, not blocking.)
  **Solution:** Noted; acceptable given the collision logic is inherently nested. Can refactor later.

P99 exceeds the 500ms guardrail — the known bottleneck is per-game SQLite moves-blob
decompression for continuation context. This is the target for optimization #4 (`encoded_move`
in DuckDB).

### Key Observations

- **Rebuild phase dominates total import time** (46s of 96s, ~48%). The rebuild re-reads every
  game from SQLite, decompresses moves, replays through chesslib, and writes to DuckDB — all
  work that Stage 1 already did and threw away.
- **Opening stats p99 is 2 seconds** — driven by a few high-fanout positions where hundreds of
  games each require a full moves-blob read + decompress from SQLite.
- **Sorted-by-zobrist gives 18× query latency improvement** over unsorted for position lookups.

## Optimization Opportunities (identified, not yet implemented)

### Import Performance

1. **Carry position rows through the pipeline (~40s savings on 1M).**
   Stage 1 (`prepare_game`) already replays every move through chesslib and has position rows in
   hand, then throws them away. After Stage 2 returns `game_id`, patch it into the position rows
   and batch-write to DuckDB. Eliminates the entire rebuild phase.

2. **Lighter rebuild read path.**
   `rebuild_position_store()` calls `store_->get(game_id)` which does a full 3-table JOIN +
   decompresses moves. A dedicated `get_moves_for_replay(game_id)` reading just the game row
   (no player/event JOIN) would cut per-game overhead.

3. **Parallel rebuild.**
   WAL mode supports concurrent readers. Split game IDs into N chunks, each thread opens its
   own SQLite connection, replays, writes to shared DuckDB buffer. ~2-3x on rebuild.

### Opening Explorer Performance

4. **Store `encoded_move USMALLINT` in DuckDB position table.**
   The move that led *to* this position. Continuations become a pure DuckDB self-join — no
   SQLite moves-blob reads at all. Eliminates the p99 tail (hundreds of game lookups).

5. **Store ECO + opening_name in DuckDB.**
   Eliminates the remaining SQLite queries. Combined with #4, `opening_stats::query()`
   becomes a pure DuckDB operation.

### Priority

| # | Change | Impact | Effort | Storage cost |
|---|---|---|---|---|
| 1 | Carry position rows through pipeline | ~40s import savings | Medium | None |
| 4 | `encoded_move` in DuckDB | Eliminates p99 tail | Medium (schema + migration) | ~160 MB |
| 5 | ECO/opening_name in DuckDB | Pure DuckDB explorer | Medium | ~200 MB |
| 2 | Lighter rebuild read | ~10-15s savings | Small | None |
| 3 | Parallel rebuild | ~15-20s savings | Medium | None |
