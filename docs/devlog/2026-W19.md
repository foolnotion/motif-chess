# Week 19 — 2026-04-30

## Decisions

- **[D040] Storage optimization: moves_hash index, zstd compression, DuckDB checkpoint.**
  Schema v2→v3 replaces the full `moves` BLOB in the unique dedup index with an 8-byte xxhash64
  hash (`moves_hash`), compresses the `moves` BLOB with zstd level 3, and adds `CHECKPOINT` after
  `sort_by_zobrist()` to reclaim DuckDB free blocks. Hash collisions are detected by comparing
  decompressed moves on the constraint-violation path, with up to 4 rehash retries. Existing v2
  databases auto-migrate on `open()`. Estimated savings: ~250 MB from the index, ~50-80 MB from
  compression, ~1.7 GB from DuckDB compaction. Rejected: wider hash (128-bit, overkill); storing
  moves uncompressed in the index (status quo, the problem we solved).

- **[D041] Carry position rows through the import pipeline to eliminate the rebuild phase.**
  `prepare_game()` now optionally emits `position_row` structs as it replays moves; stage 2 patches
  `game_id` and accumulates rows without re-reading from SQLite. Rows flush to DuckDB every 200k
  entries, bounding peak RSS to ~5 MB from the buffer (down from ~1 GB for 1M games). Measured:
  inline path saves ~35s on 1M games with only +186 MB additional peak RSS. The rebuild path is
  retained as fallback for resumed imports. Rejected: fully buffering all rows before write
  (prohibitive RSS at scale); removing the rebuild path entirely (breaks resume).

- **[D042] Store `encoded_move USMALLINT` in the DuckDB position table.**
  The chesslib 16-bit move encoding stored per position row enables pure DuckDB self-joins for the
  opening tree and opening stats, eliminating O(N_games) SQLite round-trips. `ply=0` rows store
  sentinel `0`. Storage cost: ~2 bytes × ~136M rows ≈ 272 MB at 3.4M games. Rejected: storing SAN
  strings (variable width, larger); keeping the SQLite-only approach (the p99 problem we solved).

- **[D043] Push opening_stats aggregation into a single DuckDB query.**
  Instead of iterating per-game SQLite calls, a dedicated `query_opening_stats(hash)` returns one
  row per distinct continuation with pre-aggregated counts, average elos, and sampled game IDs.
  SQLite is used only for batched ECO/opening-name attribution on sampled games — O(distinct
  continuations), not O(N matching games). Rejected: keeping the per-game loop (2s p99 on 1M);
  materialized views (unnecessary complexity at current scale).

- **[D044] Manifest dirty flag for crash-safe position rebuild.**
  A dirty flag is set on database open and cleared on clean close. If a dirty flag is detected on
  re-open, the position store is automatically rebuilt from SQLite. `game_count` is persisted to the
  manifest on close so the import pipeline can detect incomplete imports. Rejected: write-ahead
  logging for DuckDB operations (overkill — rebuild from SQLite is the reconciliation path).

- **[D045] Split game_writer from game_store to reduce coupling.**
  Write operations extracted into `game_writer` with `sqlite_impl` providing the SQLite-specific
  prepared-statement helpers. This reduces `game_store`'s cognitive complexity and isolates mutation
  logic from read logic. Struct `error` with message field and strong typedefs for `game_id` and
  `zobrist_hash` added for type safety. Rejected: keeping all operations in `game_store` (complexity
  was flagged by clang-tidy and code review).

- **[D046] Decouple from chesslib via a pimpl facade in `motif::chess`.**
  A new `source/motif/chess/` module wraps chesslib behind a stable C++ API (`parse_fen`,
  `write_fen`, `replay`, `san`, `apply_san`, `apply_uci`, `legal_moves`, `apply_encoded_move`).
  All consumers (import pipeline, engine, server, opening stats/tree) now depend on the facade
  instead of chesslib directly. The pimpl boundary isolates build times and allows swapping the
  chess backend without recompiling consumers. Rejected: keeping direct chesslib dependencies
  (header changes cascade to every consumer; no abstraction boundary).

- **[D047] Upgrade `game_id` and `zobrist_hash` from `using` aliases to proper structs.**
  The previous `using game_id = uint32_t` aliases provided no compile-time type safety — any
  `uint32_t` could be passed where a `game_id` was expected. Upgraded to struct wrappers with
  explicit constructors, `operator<=>`, and `glz::meta` specialization for transparent JSON
  serialization. Rejected: keeping `using` aliases (the original reason for D045's "strong
  typedefs" was never actually achieved); opaque strong typedef libraries (unnecessary dependency
  for two types).

## Learned

- **xxHash and zstd are already in flake.nix but not wired into CMake.** Both were pre-positioned
  as Nix buildInputs but had no `find_package`, no `target_link_libraries`, and no source includes.
  Adding them to vcpkg.json is needed for Windows/vcpkg parity (per CONVENTIONS.md: both files must
  be updated for any new dependency).

- **DuckDB `CHECKPOINT` cannot run inside a transaction.** The initial attempt embedded it in the
  `sort_by_zobrist` SQL string, which runs inside the `BEGIN`/`COMMIT` block of
  `rebuild_position_store`. Moved to a separate `position_store::checkpoint()` call after COMMIT.

- **zstd's `ZSTD_getFrameContentSize()` enables decompression without knowing the output size.**
  The compressed frame header stores the original content size (enabled by default), so
  `decompress_moves` can determine the correct allocation without a separate size field.

- **SQLite `substr()` on a BLOB doesn't work with compressed data.** The `get_continuation_contexts`
  query used `substr(g.moves, ply*2+1, 2)` to extract a 2-byte move from the raw blob. With zstd
  compression this is meaningless — had to change to reading the full blob and extracting in C++.

- **Incremental flush at 200k position rows bounds RSS to ~5 MB.** Without flushing, accumulating
  all position rows for 1M games would consume ~1 GB. A 200k-row batch is ~5 MB and can be written
  to DuckDB in a single `INSERT` without blocking the pipeline.

- **DuckDB aggregation eliminates the p99 tail for opening stats.** The per-game SQLite loop was
  dominated by a few high-fanout positions (starting position: ~1M games). A single DuckDB aggregate
  query collapses this to O(distinct continuations), which is typically < 100 rows.

- **Transposition handling in `total_games` is a subtle correctness trap.** When the same position
  appears via transpositions, `total_games` must count unique game IDs, not raw DuckDB rows. The
  initial fix overcorrected by excluding transposition counts from continuations, then required a
  revert and a second fix to include transposition-weighted frequencies while keeping unique-game
  `total_games`.

- **`gtl::flat_hash_map` is a drop-in replacement for `std::unordered_map`.** No API changes needed;
  the swap reduces memory overhead and improves cache locality. Applied across the codebase in a
  single commit.

- **Module boundary violations in `import_worker` were silently increasing coupling.** The worker
  held direct references to `game_store&` and `position_store&`, bypassing the `database_manager`
  facade. Extracting `stmt_deleter`/`unique_stmt` to a shared `sqlite_util.hpp` eliminated duplicate
  definitions across `game_store.cpp` and `schema.cpp`.

- **Glaze reflects on single-field structs as `{"value":42}`, not `42`.** When `game_id` was
  upgraded from a `using` alias to a struct with a `uint32_t value` member, HTTP JSON responses
  changed from `42` to `{"value":42}`. Fixed with a `glz::meta` specialization mapping
  `value = &T::value` so glaze treats the wrapper as transparent.

- **Pimpl facades are worth the indirection for dependency-heavy libraries.** chesslib's template-
  heavy headers added significant compile time to every consumer. The pimpl boundary in
  `motif::chess` means only `chess.cpp` recompiles when chesslib changes.

## Problems and Solutions

- **Problem:** zstd CMake target `zstd::libzstd_shared` was already defined by cpptrace's
  `Findzstd.cmake`, causing `find_package(zstd CONFIG REQUIRED)` to fail with "Some targets already
  defined."
  **Solution:** Check for existing targets before calling `find_package`, link whichever is available.

- **Problem:** `game_store::insert()` cognitive complexity exceeded 25 after adding collision
  detection loop. (clang-tidy warning, not blocking.)
  **Solution:** Noted; acceptable given the collision logic is inherently nested. Can refactor later.

- **Problem:** Opening explorer `total_games` overcounted when positions appeared via transpositions
  — raw DuckDB row count included duplicate game IDs.
  **Solution:** Changed to `COUNT(DISTINCT game_id)`, but the first fix also excluded transposition
  counts from continuation frequency. Required a revert (d34367a) and a second fix (471740f) to
  include transposition-weighted frequency while keeping unique-game `total_games`.

- **Problem:** `opening_stats::query` p99 was ~2s on 1M games, driven by per-game SQLite
  `get_continuation_contexts` calls for move extraction.
  **Solution:** Stored `encoded_move` in DuckDB and rewrote stats as a single aggregate query.
  Continuation move is now `encoded_move` from the next ply, eliminating all SQLite reads from the
  hot path.

- **Problem:** `get_game_contexts` exceeded SQLite's parameter limit (999) when called with >999
  game IDs.
  **Solution:** Batch into chunks of 999 and union results.

- **Problem:** `import_worker` held direct `game_store&` and `position_store&` references, bypassing
  the `database_manager` facade and creating a coupling boundary violation.
  **Solution:** Changed to accept `database_manager&`, accessing stores through the facade.

- **Problem:** `stmt_deleter` and `unique_stmt` were duplicated in `game_store.cpp` and `schema.cpp`.
  **Solution:** Extracted to shared `sqlite_util.hpp`.

- **Problem:** `game_id` as a struct produced `{"value":42}` in JSON instead of `42`.
  **Solution:** Added `glz::meta` specialization with `value = &T::value` to make glaze treat the
  wrapper as a transparent integer.

- **Problem:** Direct chesslib dependencies meant any chesslib header change cascaded recompilation
  to 6+ translation units.
  **Solution:** Introduced `motif::chess` pimpl facade; only `chess.cpp` now depends on chesslib
  headers.

## Open Questions

  footprint after importing with the new schema.

- Does zstd compression add meaningful latency to the import pipeline? The compress/decompress

- The `game_writer`/`sqlite_impl` split is the first step toward a cleaner db module. Should

## Baseline Benchmarks (master branch, pre-optimization)

**Machine:** Linux, Clang 21, perf-release build (`-O3 -DNDEBUG -march=x86-64-v3`)
**Corpus:** `bench/data/twic-1m.pgn` (1,001,092 games)

### Import Pipeline — 1M games

| Benchmark | Wall | Notes |
|---|---|---|
| Default fast path (4w, 64L, sorted rebuild) | 96s | Full import + rebuild + sort |
| SQLite-only serial | 50s | Floor for insert phase |
| Rebuild-only | 106s | Re-read + replay + sort |
| Rebuild ≈ total − sqlite | ~46s | Implicit from above |

| Mode | p50 (µs) | p99 (µs) | total (ms) |
|---|---|---|---|
| Unsorted | 15,761 | 42,960 | 3,357 |
| Sorted by zobrist | 882 | 22,767 | 365 |

### Opening Stats — 88M position rows, 100 sampled hashes

| Metric | Value |
|---|---|
| p50 | 868 µs |
| p99 | 2,008 ms |
| total | 3,381 ms |
| queries | 100 |
| total rows | 185 |

P99 exceeds the 500ms guardrail — the known bottleneck is per-game SQLite moves-blob
decompression for continuation context. This is the target for optimization #4 (`encoded_move`
in DuckDB).

### Key Observations

- **Rebuild phase dominates total import time** (46s of 96s, ~48%). The rebuild re-reads every
  game from SQLite, decompresses moves, replays through chesslib, and writes to DuckDB — all
  work that Stage 1 already did and threw away.
- **Opening stats p99 is 2 seconds** — driven by a few high-fanout positions where hundreds of
  games each require a full moves-blob read + decompress from SQLite.
- **Sorted-by-zobrist gives 18× query latency improvement** over unsorted for position lookups.

## Optimization Opportunities (identified, not yet implemented)

_Note: Items #1 and #4 were implemented this week — see Done section above._

### Import Performance

1. **Carry position rows through the pipeline (~40s savings on 1M).**
   Stage 1 (`prepare_game`) already replays every move through chesslib and has position rows in
   hand, then throws them away. After Stage 2 returns `game_id`, patch it into the position rows
   and batch-write to DuckDB. Eliminates the entire rebuild phase.

2. **Lighter rebuild read path.**
   `rebuild_position_store()` calls `store_->get(game_id)` which does a full 3-table JOIN +
   decompresses moves. A dedicated `get_moves_for_replay(game_id)` reading just the game row
   (no player/event JOIN) would cut per-game overhead.

3. **Parallel rebuild.**
   WAL mode supports concurrent readers. Split game IDs into N chunks, each thread opens its
   own SQLite connection, replays, writes to shared DuckDB buffer. ~2-3x on rebuild.

### Opening Explorer Performance

4. **Store `encoded_move USMALLINT` in DuckDB position table.**
   The move that led *to* this position. Continuations become a pure DuckDB self-join — no
   SQLite moves-blob reads at all. Eliminates the p99 tail (hundreds of game lookups).

5. **Store ECO + opening_name in DuckDB.**
   Eliminates the remaining SQLite queries. Combined with #4, `opening_stats::query()`
   becomes a pure DuckDB operation.

### Priority

| # | Change | Impact | Effort | Storage cost |
|---|---|---|---|---|
| 1 | Carry position rows through pipeline | ~40s import savings | Medium | None |
| 4 | `encoded_move` in DuckDB | Eliminates p99 tail | Medium (schema + migration) | ~160 MB |
| 5 | ECO/opening_name in DuckDB | Pure DuckDB explorer | Medium | ~200 MB |
| 2 | Lighter rebuild read | ~10-15s savings | Small | None |
| 3 | Parallel rebuild | ~15-20s savings | Medium | None |
