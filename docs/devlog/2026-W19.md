# Week 19 — 2026-04-23

## Decisions

- **[D010] Drop ART index, default to `sort_by_zobrist=true`:** Removed `create_zobrist_index` and `drop_zobrist_index` from `position_store`. The `rebuild_position_store` signature changed from `(bool create_index, bool sort_by_zobrist)` to `(bool sort_by_zobrist = true)`. The ART index was shown in W18 benchmarks to use ~24 GB RSS at 3.4M games while providing worse P99 than sorted-by-zobrist.

- **[D011] Seed reservoir sampling for reproducible benchmarks:** `sample_zobrist_hashes` now takes an optional `seed` parameter (default 0) and uses DuckDB's `REPEATABLE (seed)` clause. Tests pass seed 42 to eliminate run-to-run P99 variation from randomly sampling different positions.

- **[D012] Elo sourced from DuckDB, not SQLite:** `opening_move_stat` now carries `white_elo` and `black_elo` directly from the DuckDB position table, eliminating two player JOINs per query. The `get_game_contexts` batch query fetches only `(eco, opening_name, moves)` — no player data.

- **[D013] Game deduplication per continuation:** A game reaching the same Zobrist hash via transpositions (different plies) would inflate statistics. Added `(game_id, encoded_move)` deduplication via an `unordered_map` hash set.

- **[D014] Longest opening name wins:** `note_opening_name` previously kept the lexicographically smallest ECO name. Changed to prefer the longest descriptive name (e.g., "Sicilian Defense" over "Sicilian").

- **[D015] Orphaned DuckDB rows return partial stats:** If a game_id in DuckDB has no matching SQLite row, the query skips it gracefully rather than returning an error or empty stats. Also fixed the edge case where the first row was orphaned — now iterates to find the first valid row for position replay.

- **[D016] Dedicated `perf-release` preset for benchmark isolation:** Added a `perf-release` CMake preset (`-O3 -DNDEBUG -march=x86-64-v3`) separate from `release`, which carries install and packaging targets. Running benchmarks under `release` introduced noise; `perf-release` is the clean build for reproducible performance measurement. Perf assertions are enforced in `perf-release` and `release`; `dev` emits `WARN`, `dev-sanitize` skips entirely.

- **[D017] Drop non-default import strategies:** Benchmarked direct-write and partitioned-rebuild against the default fast path on 100k and 1M corpora. Direct-write was dramatically slower at every scale. Partitioned-rebuild matched speed but provided no memory advantage once made to produce correct final-table semantics. Both removed. Default fast path (`write_positions=false` + deferred sorted rebuild) is now the only supported strategy, validated on the full 3.41M-game corpus at 5:04 wall, 195% CPU, 18 GB peak RSS.

## Learned

- **DuckDB `USING SAMPLE reservoir(N ROWS) REPEATABLE (seed)` produces deterministic results.** Verified that the same seed yields identical output across runs.

- **SQLite + DuckDB disk footprint for ~818K games:** SQLite `games.db` is 371 MB (~454 bytes/game). DuckDB position table was empty in test runs because data wasn't checkpointed after `sort_by_zobrist`. Estimated DuckDB positions at ~590 MB (~40 rows/game × 19 bytes/row). For comparison, scidb's TWIC database at 3.4M games occupies 1.3 GB (~400 bytes/game).

- **P99 latency variance was caused by unseeded sampling.** Before D011, P99 varied from 321ms to 706ms between runs depending on which positions the reservoir sampler selected. With a fixed seed, results are reproducible.

- **Opening-stats p50 improved ~8% from the Elo-to-DuckDB move (D012).** Pre-review baseline on 1M games: p50=797µs, p99=116ms. Post-optimization with fixed seed: p50=730µs, p99=321–706ms depending on the sampled positions. The p99 increase relative to the initial unseeded run reflects the seeded sampler now consistently hitting the same high-fanout positions (e.g., after 1.e4) that previously caused variance. The p50 improvement is real: two `JOIN player` rounds per game are gone from the SQLite context batch. The remaining p99 bottleneck is full moves-blob deserialization for each SQLite context fetch; a future optimization could extract only `moves[2*ply]` from the blob.

- **Import strategy comparison required 1M games to be conclusive.** At 10k, differences fell within measurement noise. At 1M, direct-write and partitioned-rebuild both lost to the default fast path on every metric that matters — speed, memory, and code complexity. Removing both strategies was an easy call once the 1M numbers were in.

## Problems and Solutions

- **Problem:** P99 latency varied 321–706ms between runs due to unseeded reservoir sampling selecting different high-fanout positions.
  **Solution:** Added `seed` parameter to `sample_zobrist_hashes` using DuckDB `REPEATABLE (seed)`.

- **Problem:** Orphaned DuckDB rows (game_id with no SQLite match) caused empty stats when the first row was orphaned.
  **Solution:** Iterate through `opening_moves` to find the first valid row for position replay instead of using `front()`.

- **Problem:** `get_opening_contexts` JOINed the `player` table twice for white/black Elo, adding per-game overhead.
  **Solution:** Replaced with `get_game_contexts` that fetches only `(eco, opening_name, moves)`. Elo now comes from DuckDB `opening_move_stat`.

- **Problem:** Transposition duplicates inflated continuation counts — a game reaching the same Zobrist hash via different plies was counted multiple times per continuation.
  **Solution:** Added `(game_id, encoded_move)` hash-set deduplication.

- **Problem:** `note_opening_name` kept lexicographically smallest name, so "Sicilian" won over "Sicilian Defense".
  **Solution:** Changed to keep the longest name.

- **Problem:** Import strategy benchmark at 10k games was inconclusive — differences between direct-write, partitioned-rebuild, and default fast path fell within noise.
  **Solution:** Re-ran on 100k and 1M corpora. Default fast path won decisively at both scales; alternatives removed.
