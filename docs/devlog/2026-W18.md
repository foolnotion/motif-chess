# Week 18 — 2026-04-30

## Decisions

- **[D009] Drop ART index in favor of sorted-by-zobrist + zonemap pruning:** Query latency
  benchmarking at 288M position rows (3.4M games) showed sorted no-index dominates the ART
  index: P50 1,254us vs 630us, but P99 58ms vs 224ms, total 608ms vs 2,428ms, and RSS
  ~3 GB vs ~24 GB. The ART index's high variance comes from memory pressure on the massive
  index structure. Zobrist-sorted table gives near-index P50, 4x better P99, and zero index
  memory overhead. Unsorted no-index is a full table scan at 50ms P50 — dead on arrival at
  scale. Rejected: keeping the ART index (prohibitive memory at 10M games); unsorted table
  (too slow); partitioned indexes (unnecessary given zonemap performance).

- **[D035] Store the starting position (`ply=0`) for every non-empty game:** Position counts
  and opening statistics need an explicit row for the initial board state. Rejected: inferring
  the start position from `game_count`, because deletion, rebuild, and search code already use
  the DuckDB position table as the source of truth.

- **[D036] Reject zero-move PGN games at import time:** Header-only games created a mismatch
  between SQLite game count and opening statistics because they had metadata but no valid first
  continuation. Added a distinct `empty_game` import error and rejected them in both direct game
  creation and bulk import paths. Rejected: storing them as start-position-only games, because
  they pollute opening statistics without contributing chess content.

- **[D037] `opening_stats.total_games` means unique stored games that reached the queried position:**
  Raw DuckDB row count was wrong in the presence of orphan rows, duplicate rows, and terminal
  positions. The query now counts unique SQLite-backed game IDs from the matching position hits,
  including games where the position is final and has no continuation. Rejected: counting only
  continuation-bearing games, because terminal positions are still legitimately reached by games.

- **[D038] Pre-commit must fail hard on `clang-tidy` errors and use the project compile database:**
  The original hook swallowed tidy failures with `|| true`, which allowed broken commits to land.
  The tracked hook script now requires `build/dev/compile_commands.json` and runs `clang-tidy -p build/dev`.
  Rejected: leaving the hook best-effort only, because it creates false confidence during local review.

- **[D039] Release publishing is tag-driven via Quay and GitHub Actions:** The first release uses
  annotated tag `v0.1.0`, which triggers `.github/workflows/publish.yml` and pushes both
  `quay.io/motif-chess/motif-server:v0.1.0` and `:latest`. Rejected: manual Docker pushes,
  because reproducible release publication belongs in CI.

## Learned

- **Zobrist sort + zonemaps eliminates the ART index at DuckDB scale:** DuckDB's zonemap
  pruning on a `zobrist_hash`-sorted `position` table delivers P50 within 2x of the ART
  index while using 8x less memory and having a tighter P99 tail. The index was masking
  its own cost: at 288M rows the index structure dominated RSS and caused high-latency
  outliers from cache pressure.

- **Query latency is NOT the problem at current scale:** Even the slowest tested variant
  (ART indexed, P99 224ms) would still meet spec 003's position search target (< 10ms for
  1M games) at the 1M scale. The real win is eliminating the 21.6 GB RSS spike during
  index creation, which was the memory dealbreaker identified in W17.

- **Full-scan latency scales predictably:** Unsorted no-index at 288M rows: 50ms P50.
  Extrapolating to 10M games (~850M rows): ~150ms per query — still within NFR01's 100ms
  P99 if we sort. At aix scale (7B games), parallel scanning would be needed.

- **SAN A/B at 10k is representative:** The `replayer` vs `from_string` comparison at 10k
  (1.10s vs 1.06s) showed near-parity, consistent with the W17 finding that SAN parsing
  is no longer the bottleneck.

- **Opening-stat consistency failures split into two classes:** start-position undercounts came
  from missing `ply=0` rows, while overcounts came from orphaned DuckDB rows and ambiguous
  `total_games` semantics. The same symptom (`games/count` mismatch) can come from different
  layers and needs targeted tests.

- **Terminal positions are an easy trap for opening APIs:** deriving total-game count from
  continuations silently drops games where the queried position is the last position in the game.
  A separate regression test is required for final-ply positions.

- **A green pre-commit message is meaningless if the hook masks failures:** local tooling must
  exit non-zero on analysis errors or engineers will trust a result that never actually enforced
  the contract.

## Problems and Solutions

- **Problem:** No query method existed in `position_store` — only insert and schema ops.
  **Solution:** Added `query_by_zobrist`, `query_opening_moves`, and
  `sample_zobrist_hashes` using the DuckDB C API (`duckdb_query`, `duckdb_value_*`).

- **Problem:** Sample hash collection for benchmarking required accessing the DuckDB
  connection, which is internal to `position_store`.
  **Solution:** Added `sample_zobrist_hashes` using DuckDB's `USING SAMPLE reservoir(N)`
  SQL syntax — clean, random, and doesn't require chesslib replay in the test.

- **Problem:** Benchmark initially hardcoded the 10k PGN path, so `MOTIF_IMPORT_PERF_PGN`
  override was ignored.
  **Solution:** Changed to use `perf_pgn_path()` from the test helper.

- **Problem:** `/api/games/count`, start-position `total_games`, and continuation sums drifted
  apart in live data.
  **Solution:** Added `ply=0` rows for all non-empty games, rejected zero-move imports, and changed
  `total_games` to count unique valid game IDs instead of raw DuckDB row count or continuation count.

- **Problem:** deleting one manual game rebuilt the entire DuckDB position store and still left
  cleanup semantics hard to reason about.
  **Solution:** Added `delete_by_game_id()` and switched single-game delete to targeted row removal.

- **Problem:** pre-commit claimed success even when `clang-tidy` failed to resolve headers.
  **Solution:** removed failure masking, required `build/dev/compile_commands.json`, and used
  `clang-tidy -p build/dev`.

- **Problem:** `master` CI was red on a trivial README spelling regression.
  **Solution:** corrected the `codespell` failure (`Analyse` -> `Analyze`) and repushed.

## Open Questions

- Should `rebuild_position_store` default to `sort_by_zobrist=true` and
  `create_index=false` going forward? The benchmark strongly supports this, but it
  changes the default behavior for all callers.

- What is the query latency profile on the 1M corpus (~85M position rows)? This would
  validate the scaling curve between 810k and 288M rows.

- Prepared statements vs string-concatenated SQL: the current `query_by_zobrist`
  builds a new SQL string per query. DuckDB prepared statements (`duckdb_prepare`)
  would eliminate string construction overhead and may further tighten the sorted
  variant's P50.

- Cross-database atomicity is still unresolved: SQLite and DuckDB updates can still diverge on
  crashes or partial failures outside rebuild flows. Do we want a reconciliation command beyond
  `POST /api/positions/rebuild`, or periodic consistency checks?

- Should we add CI coverage for the release workflow path (for example, a dry-run image build on
  branch pushes) so Quay publication failures are caught before tagging?
